## Decisions Made
- `find_devices_no_probe()` reads USB descriptors only, never opens sessions or claims interfaces -- this prevents the daemon's discovery from disrupting IF2 (used by bridge) or triggering STALL recovery resets
- Initial connect uses exact bus/address from discovery for precision; reconnect uses VID/PID since USB address changes after replug
- Consecutive error threshold (10) detects disconnect when kernel returns ambiguous errors (Pipe, Other) instead of explicit NoDevice/Io
- All errors in the main loop trigger retry with 2s settle delay instead of daemon exit -- the daemon should never give up

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] probe_devices disrupted bus during InputOnly discovery**
- **Found during:** Task 2 (human verification)
- **Issue:** `enumerate_devices` calls `probe_devices` which opens IF2, sends vendor commands, and performs STALL recovery. This disrupted the input path when the daemon was trying to claim IF0 only.
- **Fix:** Added `find_devices_no_probe()` that matches VID/PID from USB descriptors without opening any sessions. Changed daemon to use it.
- **Files modified:** `crates/monsgeek-transport/src/discovery.rs`, `crates/monsgeek-inputd/src/daemon.rs`
- **Verification:** Daemon starts cleanly without bus disruption
- **Committed in:** `64735b6`

**2. [Rule 1 - Bug] Daemon exited on non-disconnect errors**
- **Found during:** Task 2 (human verification)
- **Issue:** The main loop only retried on `DaemonError::Disconnected`. Any other error (transient USB failures, kernel quirks) caused the daemon to exit permanently.
- **Fix:** Changed outer loop to retry on all errors with 2s settle delay. Only udev wait is triggered for explicit disconnects.
- **Files modified:** `crates/monsgeek-inputd/src/daemon.rs`
- **Verification:** Daemon survives transient errors during testing
- **Committed in:** `64735b6`

**3. [Rule 1 - Bug] Kernel returns ambiguous errors on unplug**
- **Found during:** Task 2 (human verification)
- **Issue:** Some Linux kernels return `Pipe` or `Other` instead of `NoDevice`/`Io` when the USB device is unplugged. The explicit `TransportError::Disconnected` match missed these cases, leaving the daemon polling a dead device.
- **Fix:** Added consecutive error counter (threshold 10). If 10 reads fail without a single success, treat as disconnect.
- **Files modified:** `crates/monsgeek-inputd/src/daemon.rs`
- **Verification:** Disconnect/reconnect cycle works cleanly during testing
- **Committed in:** `64735b6`

**4. [Rule 1 - Bug] USB address changes after replug**
- **Found during:** Task 2 (human verification)
- **Issue:** After unplug/replug, the USB address changes. The daemon was using VID/PID for all connections via `open_with_mode`, but the initial connect should use the exact address from discovery for precision.
- **Fix:** Initial connect uses `open_at_with_mode(bus, addr)`. After disconnect, `use_location` is set to `None` and reconnect falls back to `open_with_mode(vid, pid)`.
- **Files modified:** `crates/monsgeek-inputd/src/daemon.rs`
- **Verification:** Initial connect and reconnect both work correctly
- **Committed in:** `64735b6`
