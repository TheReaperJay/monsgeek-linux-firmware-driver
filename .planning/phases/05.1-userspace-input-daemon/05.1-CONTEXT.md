# Phase 5.1: Userspace Input Daemon - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Persistent daemon binary that claims IF0 from the kernel, reads raw HID boot protocol reports, applies software debounce and correct key ordering, and injects cleaned events via uinput — eliminating compositor jitter and switch bounce from the keyboard input path. Runs as a separate binary with independent lifecycle from the gRPC bridge. Does not modify the bridge or its IF2 vendor command path.

</domain>

<decisions>
## Implementation Decisions

### Daemon binary
- Standalone binary (`monsgeek-inputd`), independent of the gRPC bridge
- Designed to run as a systemd service
- One daemon instance per keyboard (multiple keyboards = multiple instances, systemd template unit)
- Uses `sd_notify` for lifecycle signaling: `READY=1` after uinput device is created, `WATCHDOG=1` for health, `STATUS=` for diagnostics (e.g., "Connected to M5W (1308)", "Waiting for keyboard")
- Signal handling: `SIGTERM`/`SIGINT` trigger clean shutdown — release all held keys via uinput, destroy uinput device, release IF0/IF1, reattach kernel drivers. Non-negotiable because unclean exit leaves the keyboard dead.

### Disconnect/reconnect behavior
- On keyboard disconnect: release all held keys via uinput, destroy the uinput device, enter wait state polling for keyboard reappearance via udev
- On keyboard reconnect: recreate USB session, recreate uinput device, resume input processing
- Daemon stays alive across disconnect/reconnect cycles — does not exit

### uinput device identity
- Distinct identity from the physical keyboard — custom name (e.g., "MonsGeek M5W (monsgeek-inputd)"), not mirroring physical VID/PID
- Physical device's IF0 is detached from kernel, so the uinput device is the only keyboard the desktop environment sees
- Distinct naming makes debugging (evtest, libinput) and per-device configuration unambiguous

### Coexistence model — independent USB sessions
- Daemon and bridge are separate processes with separate `libusb` handles to the same USB device
- Daemon claims IF0/IF1 (input interfaces), bridge claims IF2 (vendor commands)
- USB interfaces are independent endpoints by design — no coordination needed
- Requires new `SessionMode::InputOnly` variant that claims IF0/IF1 but NOT IF2 (current `UserspaceInput` claims all three)
- Bridge code does not change at all — it keeps using `TransportOptions::control_only()` as-is
- Independent failure domains: bridge crash doesn't affect typing, daemon crash doesn't affect configurator

### Debounce configuration
- CLI flag `--debounce-ms <value>` overrides default
- Default value lives in a config file (`/etc/monsgeek/inputd.conf` or `~/.config/monsgeek/inputd.conf`)
- Priority hierarchy: CLI flag > config file > hardcoded 15ms fallback
- 15ms default is safely above the 6-12ms measured spacebar bounce, low enough to be imperceptible
- This is SOFTWARE debounce only — stacks on top of firmware debounce (1ms on-chip via SET_DEBOUNCE), catches the mechanical bounce that firmware doesn't filter
- Firmware debounce remains independently configured through the web configurator via the bridge on IF2

### Software vs firmware debounce
- Firmware debounce: on-chip, configured via SET_DEBOUNCE, filters before HID reports are sent. M5W default is 1ms — too weak for 6-12ms switch bounce.
- Software debounce: in the daemon, applied after receiving HID reports. Tracks per-key release timestamps, suppresses re-press within the debounce window.
- Flow: physical switch → firmware debounce (1ms) → HID report → daemon IF0 → software debounce (15ms) → uinput → desktop
- Software layer also fixes key ordering (releases before presses) which firmware doesn't do

### Claude's Discretion
- Config file format (TOML, simple key=value, etc.)
- Exact uinput device name string
- Log verbosity levels and format
- udev reconnect polling interval
- Whether to support `--device` flag for multi-keyboard targeting or auto-detect

</decisions>

<specifics>
## Specific Ideas

- The transport layer's `pump_input` and `InputProcessor` do most of the heavy lifting already — the daemon is primarily: open device in InputOnly mode, create uinput device, bridge TransportEvent::InputActions to uinput writes, handle lifecycle
- Measured data: Mutter compositor adds 342us p50 with 2.5-18ms jitter at p95; switch bounce is 6-12ms through 1ms firmware debounce. The daemon bypasses the compositor's input processing entirely by injecting via uinput.
- Config file should be simple — there's currently one tunable parameter (debounce_ms). Don't over-engineer the config format.

</specifics>
