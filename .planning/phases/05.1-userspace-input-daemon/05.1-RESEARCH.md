# Phase 05.1: Userspace Input Daemon - Research

**Researched:** 2026-03-26
**Domain:** Linux uinput virtual keyboard, systemd daemon lifecycle, USB HID boot protocol input processing
**Confidence:** HIGH

## Summary

This phase builds a standalone daemon binary (`monsgeek-inputd`) that claims IF0/IF1 from the kernel, reads raw HID boot protocol reports via the existing transport layer, applies software debounce and key ordering corrections, and injects cleaned events via a uinput virtual keyboard device. The transport layer already provides `InputProcessor` (with 14 passing tests), `pump_input()`, IF0 claiming, SET_PROTOCOL(Boot), keymap tables, and `all_keycodes()` for uinput registration. The daemon is primarily glue: open device in a new `SessionMode::InputOnly`, create uinput device, bridge `KeyAction` events to uinput `emit()` calls, handle lifecycle (disconnect/reconnect, signal handling, systemd notification).

The `evdev` crate (0.13.2) provides the cleanest uinput API with `VirtualDeviceBuilder` and automatic `SYN_REPORT` batching. Its `KeyCode` is a newtype around `u16` with `KeyCode::new(code)` constructor, which maps directly to the Linux keycodes already defined in `keycodes.rs`. The `sd-notify` crate (0.5.0) provides lightweight systemd `READY=1`, `WATCHDOG=1`, and `STATUS=` notifications without pulling in libsystemd. CLI argument parsing uses `clap` (4.6.0) with derive macros.

**Primary recommendation:** Use `evdev` 0.13.2 for uinput, `sd-notify` 0.5.0 for systemd integration, `clap` 4.6.0 for CLI, and `toml` 1.1.0 for config file parsing. Create a new binary crate `monsgeek-inputd` in the workspace. Add `SessionMode::InputOnly` to `usb.rs` that claims IF0/IF1 but NOT IF2.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Standalone binary (`monsgeek-inputd`), independent of the gRPC bridge
- Designed to run as a systemd service (systemd template unit for multi-keyboard)
- Uses `sd_notify` for lifecycle signaling: `READY=1` after uinput device created, `WATCHDOG=1` for health, `STATUS=` for diagnostics
- Signal handling: `SIGTERM`/`SIGINT` trigger clean shutdown -- release all held keys via uinput, destroy uinput device, release IF0/IF1, reattach kernel drivers
- On disconnect: release all held keys, destroy uinput device, enter wait state polling for keyboard via udev
- On reconnect: recreate USB session, recreate uinput device, resume input processing
- Daemon stays alive across disconnect/reconnect cycles
- Distinct uinput device identity from physical keyboard -- custom name, not mirroring physical VID/PID
- Daemon and bridge are separate processes with separate libusb handles
- Daemon claims IF0/IF1 (input), bridge claims IF2 (vendor commands)
- Requires new `SessionMode::InputOnly` variant that claims IF0/IF1 but NOT IF2
- Bridge code does not change
- CLI flag `--debounce-ms <value>` overrides default
- Default value from config file (`/etc/monsgeek/inputd.conf` or `~/.config/monsgeek/inputd.conf`)
- Priority hierarchy: CLI flag > config file > hardcoded 15ms fallback
- Software debounce only -- stacks on top of firmware debounce

### Pattern 1: SessionMode::InputOnly
**What:** New variant in `usb.rs::SessionMode` that claims IF0/IF1 but NOT IF2.
**When to use:** When the daemon needs keyboard input without vendor command access.
**Implementation:**

The existing `SessionMode::UserspaceInput` claims IF0/IF1/IF2. The new `InputOnly` variant modifies the interface list to `&[IF0, IF1]` only. The open/drop logic already handles per-interface detach/reattach based on which interfaces were claimed.

Key changes in `usb.rs`:
- Add `SessionMode::InputOnly` variant
- `open_impl()`: match on `InputOnly` to claim `&[IF0, IF1]` (not IF2)
- `Drop`: release IF0/IF1 only (not IF2)
- `ep_in` is set for both `UserspaceInput` and `InputOnly`
- SET_PROTOCOL(Boot) sent for both modes

### Pattern 2: uinput Device Lifecycle
**What:** Create/destroy uinput device tied to USB session lifecycle.
**When to use:** Every connect/disconnect cycle.
**Flow:**
```
startup:
  1. Find keyboard via discovery (or --device flag)
  2. Open UsbSession with InputOnly mode
  3. Create VirtualDevice via evdev builder
  4. sd_notify READY=1
  5. Enter poll loop

poll loop:
  1. Read IF0 interrupt endpoint (10ms timeout)
  2. Process report through InputProcessor
  3. Convert KeyAction to evdev InputEvent
  4. Emit via VirtualDevice::emit()

disconnect:
  1. InputProcessor::release_all_keys()
  2. Emit release events via uinput
  3. Drop VirtualDevice (destroys uinput node)
  4. Drop UsbSession (releases IF0/IF1, reattaches kernel)
  5. sd_notify STATUS="Waiting for keyboard"
  6. Poll udev for reconnect

reconnect:
  1. Re-open UsbSession with InputOnly mode
  2. Recreate VirtualDevice
  3. sd_notify STATUS="Connected to M5W (1308)"
  4. Resume poll loop

shutdown (SIGTERM/SIGINT):
  1. InputProcessor::release_all_keys()
  2. Emit release events via uinput
  3. Drop VirtualDevice
  4. Drop UsbSession
  5. Exit cleanly
```

### Pattern 3: KeyAction to evdev InputEvent Conversion
**What:** Map `KeyAction { keycode: u16, value: i32 }` to evdev `InputEvent`.
**Implementation:**
```rust
// KeyAction.keycode is already a Linux keycode (from keycodes.rs)
// KeyCode::new() accepts a raw u16
// EventType::KEY.0 == 1 (EV_KEY)
// evdev::InputEvent::new(type_, code, value) -- zero timestamp, kernel fills it

fn key_action_to_input_event(action: &KeyAction) -> InputEvent {
    InputEvent::new(EventType::KEY.0, action.keycode, action.value)
}

// VirtualDevice::emit() auto-appends SYN_REPORT
// So a batch of KeyActions becomes:
//   device.emit(&events)?;  // one call, SYN_REPORT added automatically
```

### Anti-Patterns to Avoid
- **Sharing UsbSession between daemon and bridge:** USB interfaces are independent endpoints. Separate processes, separate `rusb` contexts. No IPC, no coordination, no shared state.
- **Mirroring physical device VID/PID on uinput:** Creates confusion in `evtest`/`libinput` about which device is real vs virtual. Use a distinct name.
- **Keeping uinput device alive across USB disconnects:** The uinput device must be destroyed and recreated on each reconnect cycle. Stale uinput devices will have stuck keys.
- **Polling for reconnect without udev:** Busy-looping on `rusb::devices()` wastes CPU. Use udev monitor with `poll()` (pattern already exists in `thread.rs::spawn_hotplug_thread`).
- **Blocking the input poll loop:** The 10ms poll timeout drives input latency. Config loading, udev setup, and sd_notify calls must happen outside the hot path.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| uinput device creation | Raw ioctl calls to /dev/uinput | `evdev::uinput::VirtualDeviceBuilder` | Builder handles UI_DEV_SETUP, set_evbit, set_keybit ioctls; `emit()` auto-appends SYN_REPORT |
| SYN_REPORT batching | Manual SYN_REPORT after every key event | `VirtualDevice::emit(&[events])` | evdev auto-terminates each emit batch with SYN_REPORT |
| HID report processing | Custom boot protocol parser | `InputProcessor` in `input.rs` | Already written, 14 tests, handles debounce, ordering, modifiers |
| Linux keycode mapping | Manual HID-to-Linux table | `HID_TO_LINUX` / `MODIFIER_KEYCODES` in `keymap.rs` | Hardware-verified, 100+ entries, tested against Python reference |
| Signal handling | Raw libc signal handlers | `signal-hook` crate | Safe Rust signal handling without unsafe blocks |
| systemd notification | Raw socket writes to $NOTIFY_SOCKET | `sd-notify` crate | Handles socket path, message formatting, graceful degradation when not under systemd |
| USB hot-plug detection | libusb hotplug callbacks | udev monitor with `poll()` | libusb hotplug unreliable on Linux (verified empirically in this project) |

**Key insight:** The transport layer already contains 80% of the input processing logic. The daemon is a thin lifecycle wrapper around existing, tested infrastructure. The risk is in lifecycle management (disconnect/reconnect, signal handling, uinput creation/teardown), not in the core input path.

## Common Pitfalls

### Pitfall 1: Stuck Keys on Disconnect
**What goes wrong:** Daemon loses USB connection mid-keypress. uinput device still reports key as held. User's typing is permanently broken until they physically press and release the stuck key.
**Why it happens:** No release events were sent before the USB session died.
**How to avoid:** On ANY session teardown path (disconnect, shutdown, error), call `InputProcessor::release_all_keys()` and emit the resulting release events through uinput BEFORE destroying the uinput device.
**Warning signs:** Key repeat running infinitely after unplugging keyboard.

### Pitfall 2: uinput Device Not Destroyed on Crash
**What goes wrong:** Daemon crashes or gets SIGKILL. uinput device node persists as a ghost device. Desktop environment sees two keyboards on next daemon start.
**How to avoid:** SIGKILL cannot be caught, so this is an edge case. On startup, check for stale uinput devices (by name) and log a warning. The Drop impl on `VirtualDevice` handles normal cleanup. For SIGKILL, the kernel cleans up when the file descriptor closes (process exit closes all fds), so this is actually handled by the OS.
**Warning signs:** Duplicate keyboard entries in `libinput list-devices`.

### Pitfall 3: Race Between Daemon and Bridge for IF0
**What goes wrong:** Both daemon and bridge try to claim IF0. The second process gets `EBUSY`.
**Why it happens:** Bridge uses `SessionMode::ControlOnly` which claims only IF2. But if someone accidentally changes bridge to claim IF0, or if the `InputOnly` mode accidentally includes IF2, the processes conflict.
**How to avoid:** `SessionMode::InputOnly` claims exactly `&[IF0, IF1]`. `SessionMode::ControlOnly` claims exactly `&[IF2]`. These are non-overlapping sets by construction. The bridge binary does not change.
**Warning signs:** `rusb::Error::Busy` during `claim_interface`.

### Pitfall 4: SET_PROTOCOL(Boot) Not Sent After Reconnect
**What goes wrong:** Keyboard reconnects but stays in Report Protocol mode. Reports are in a format the daemon doesn't parse. All keypresses are garbled or lost.
**Why it happens:** When using HID_QUIRK_IGNORE, no kernel driver probes IF0, so SET_PROTOCOL(Boot) is never sent by the kernel. The daemon's `open_impl()` handles this, but only if the `SessionMode` triggers the `claims_input` path.
**How to avoid:** `SessionMode::InputOnly` must set `claims_input = true` so that `open_impl()` sends SET_PROTOCOL(Boot). This is already the pattern for `UserspaceInput`.
**Warning signs:** `process_report()` returns empty actions for every report, or reports are consistently < 8 bytes.

### Pitfall 5: Debounce State Persists Across Disconnect
**What goes wrong:** Old debounce release timestamps from before disconnect cause legitimate keypresses after reconnect to be suppressed.
**Why it happens:** `InputProcessor` tracks `release_times` HashMap. If the processor instance persists across a disconnect/reconnect cycle, stale timestamps remain.
**How to avoid:** Create a fresh `InputProcessor` on each reconnect. Or call `release_all_keys()` which clears `release_times`.
**Warning signs:** Keys being silently suppressed after keyboard replug.

### Pitfall 6: Config File Permissions
**What goes wrong:** Daemon runs as a non-root user (via systemd) but config file at `/etc/monsgeek/inputd.toml` is root-only readable.
**How to avoid:** Config loading should gracefully fall back if file is unreadable (log warning, use defaults). User-level config at `~/.config/monsgeek/inputd.toml` takes priority and is always readable by the running user.
**Warning signs:** Daemon always using hardcoded defaults despite config file existing.

## Code Examples

### Creating the uinput Virtual Device
```rust
// Source: evdev 0.13.2 docs -- VirtualDeviceBuilder API
use evdev::uinput::VirtualDeviceBuilder;
use evdev::{AttributeSet, EventType, InputEvent, KeyCode};
use monsgeek_transport::keymap::all_keycodes;

fn create_uinput_device() -> std::io::Result<evdev::uinput::VirtualDevice> {
    let mut keys = AttributeSet::<KeyCode>::new();
    for keycode in all_keycodes() {
        keys.insert(KeyCode::new(keycode));
    }

    VirtualDeviceBuilder::new()?
        .name("MonsGeek M5W (monsgeek-inputd)")
        .with_keys(&keys)?
        .build()
}
```

### Emitting Key Events
```rust
// Source: evdev 0.13.2 docs -- VirtualDevice::emit() auto-appends SYN_REPORT
use evdev::{EventType, InputEvent};
use monsgeek_transport::input::KeyAction;

fn emit_actions(
    device: &mut evdev::uinput::VirtualDevice,
    actions: &[KeyAction],
) -> std::io::Result<()> {
    if actions.is_empty() {
        return Ok(());
    }
    let events: Vec<InputEvent> = actions
        .iter()
        .map(|a| InputEvent::new(EventType::KEY.0, a.keycode, a.value))
        .collect();
    device.emit(&events)
}
```

### Config File
```rust
// Source: toml 1.1.0 + serde
#[derive(serde::Deserialize, Default)]
struct Config {
    debounce_ms: Option<u64>,
}

fn load_config() -> Config {
    let user_config = dirs_next::config_dir()
        .map(|d| d.join("monsgeek/inputd.toml"));
    let system_config = std::path::Path::new("/etc/monsgeek/inputd.toml");

    // User config takes priority
    if let Some(path) = &user_config {
        if let Ok(contents) = std::fs::read_to_string(path) {
            if let Ok(config) = toml::from_str::<Config>(&contents) {
                return config;
            }
        }
    }

    // Fall back to system config
    if let Ok(contents) = std::fs::read_to_string(system_config) {
        if let Ok(config) = toml::from_str::<Config>(&contents) {
            return config;
        }
    }

    Config::default()
}
```

## Open Questions

1. **`dirs-next` or `dirs` crate for XDG config path?**
   - What we know: Both provide `config_dir()` for `~/.config`. `dirs-next` is a maintained fork.
   - Recommendation: Use `dirs-next` (0.2.0) or just hardcode `$HOME/.config/monsgeek/inputd.toml` since this is the only config path needed. Hardcoding avoids adding a dependency for one function call.

2. **Should the daemon auto-detect the keyboard or require `--device`?**
   - What we know: The existing `open_matching_session()` in `lib.rs` already handles device discovery by firmware ID. Single-keyboard case is the common path.
   - Recommendation: Auto-detect by default using `DeviceRegistry` + `open_matching_session()`. Support `--device bus:addr` for explicit targeting when multiple keyboards are connected. This matches the bridge's discovery model.

3. **udev reconnect polling interval**
   - What we know: The existing `spawn_hotplug_thread` uses `poll()` with 1-second timeout. This is the same pattern needed for reconnect.
   - Recommendation: Reuse the same udev monitor pattern. 1-second poll timeout is fine -- reconnect is not latency-sensitive.
